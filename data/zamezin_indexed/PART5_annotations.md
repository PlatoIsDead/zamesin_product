# Аннотации — ЧАСТЬ 5: Книга «Как делать продукт» (Ваня Замезин)

## Глава 1 — Главное про Jobs To Be Done
**Источник:** data/book_zamesin.txt
**Тип:** Глава книги «Как делать продукт»

**Главная идея:**
Введение в теорию Jobs To Be Done. Работа — главная единица анализа. Люди не покупают продукты, они нанимают решения для выполнения своих задач. Сегментация по работам — главная сегментация для создания продукта.

**Ключевые темы:**
- **Работа как единица анализа:** Человек в точке А хочет перейти в точку Б. Всё, что помогает этому переходу — решение. Работа = цель, сформированная мозгом для удовлетворения потребности.
- **«Нанять решение на работу»:** Когда человек выбрал продукт — он нанял его на работу. Выбирает из известных ему решений (Consideration Set).
- **Работа первична, решения заменяемы:** Клиенту Т-Банка не нужен счёт и приложение — он хочет спокойно спать и переводить деньги. Продукт — лишь лучшее известное решение.
- **Consideration Set:** Список всех известных человеку способов выполнить работу. Наш продукт побеждает, если известен и воспринимается ценнее конкурентов.
- **Сегментация по работам:** Демографическая сегментация (возраст, доход) не объясняет причинно-следственные связи. Сегментация по парам работа+решение даёт алгоритмы действий.
- **Чинить боли — не единственный способ:** Проблема — это следствие того, что решение облажалось на работе. На рынке могут быть сегменты без проблем, но с возможностью создать ценность.
- **JTBD применим для B2B:** Работы есть у всех ролей в сделке (ЛПР, конечный пользователь, бизнес-заказчик), включая личные работы стейкхолдеров.

**Концепции / инструменты:**
Работа, Решение, «Нанять на работу», Consideration Set, Сегментация по работам, Точка А → точка Б, Демография vs работы, B2B-специфика

## Глава 2 — Введение в Advanced Jobs To Be Done
**Источник:** data/book_zamesin.txt
**Тип:** Глава книги «Как делать продукт»

**Главная идея:**
AJTBD — практическое расширение JTBD, которое даёт алгоритмы решения бизнес-задач. Граф работ — вторая единица анализа. Ценность = более энергоэффективно выполнять работы. Из граф работ вытекают все механики создания ценности и стратегия продукта.

**Ключевые темы:**
- **Почему AJTBD, а не просто JTBD:** JTBD — гениальная, но теоретическая теория без алгоритмов. AJTBD добавляет: граф работ, принципы решения мозгом, алгоритмы создания ценности, алгоритмы бизнес-задач.
- **Граф работ:** Иерархия работ разного уровня (Big Job → Core Job → Small Jobs). Реальный граф формируется двумя факторами: строением решения и выбором человека. Из знания графа рождается ценность и стратегия.
- **Что такое ценность:** Ценность = более энергоэффективно выполнять работы (меньше инвестиций, больше пользы, относительно ожиданий мозга). Базовое качество = попасть в предсказание мозга о костах и пользе.
- **Как ценность вытекает из графа:** Все успешные решения объясняются оптимизацией графов работ. Механики: починить проблему, выполнять больше работ одним решением, убить работы (упрощение), выйти на предыдущую/следующую работу.
- **50+ механик решения бизнес-задач:** Каждая бизнес-задача (запуск, удержание, рост чека, конверсия, выход из конкуренции) имеет алгоритм применения механик над графами сегментов.
- **Алгоритм discovery:** 1) Определить бизнес-задачу. 2) Построить гипотезы графов сегментов. 3) Предположить механики. 4) Провести исследование. 5) Применить механики.
- **Соотношение с другими фреймворками:** Граф работ — CJM на стероидах. ICP — дополнительные критерии к работам. UX = механики убийства работ и починки проблем. AAARRR объясняется через работы.
- **Границы AJTBD:** Не работает для аддиктивных продуктов, травматического поведения, неосознаваемых сильных потребностей (статус, продолжение рода).

**Концепции / инструменты:**
Граф работ, Big Job, Core Job, Small Job, Ценность как энергоэффективность, Предсказательная функция мозга, 50+ механик, Алгоритм бизнес-задач, AJTBD vs JTBD

## Глава 3 — Почему проблема это проблема
**Источник:** data/book_zamesin.txt
**Тип:** Глава книги «Как делать продукт»

**Главная идея:**
Проблема — это не боль сама по себе. Это следствие того, что на работу было нанято решение и оно облажалось: не выполнило работу или выполнило ниже ожиданий. Понимание структуры проблемы позволяет принимать более точные продуктовые решения.

**Ключевые темы:**
- **Проблема = решение облажалось на работе:** Всегда есть работа → нанято решение → решение выполнило работу ниже ожиданий. Проблемы не существуют сами по себе.
- **Что происходит в мозге при проблеме:** Негативные эмоции (злость, раздражение) как сигнал «важная цель не выполнена». Просадка дофамина от ошибки предсказания. Вынужденные налоговые работы.
- **Почему проблема — угроза для бизнеса:** Каждая проблема переобучает мозг «не использовать это решение». Риск потери клиента, негативных отзывов, роста CAC.
- **Почему проблема — возможность для бизнеса:** Если есть большой сегмент с сильной проблемой с текущим решением — и вы умеете её починить — это самая высокая конверсия в переключение.
- **Ошибки при починке проблем:** Чинить проблемы не тех клиентов (не маржинальных, не фокусных). Чинить редкие проблемы. Чинить вне критической последовательности. Сводить всё создание ценности только к починке проблем.
- **Проблемы в критических последовательностях:** Приоритет всегда — проблемы в критической последовательности работ (не устранив — клиент уйдёт).
- **Как извиняться:** Лучшее извинение — выполнить исходную работу без проблем, а не дарить бонусы.

**Концепции / инструменты:**
Проблема, Налоговые работы, Предсказательная модель мозга, Дофамин, Критическая последовательность работ, Починка проблем как механика ценности

## Глава 4 — Невероятная сила привычки
**Источник:** data/book_zamesin.txt
**Тип:** Глава книги «Как делать продукт»

**Главная идея:**
Привычка — эволюционный механизм экономии ресурсов мозга. Под каждый продукт/интерфейс формируются нейросети автоматического поведения. Это объясняет, почему люди не переключаются на лучшие решения и как заставить их переключиться.

**Ключевые темы:**
- **Что такое привычка:** Физическое изменение структуры мозга, позволяющее автоматически выполнять рутину. Пользователи ожидают привычного поведения интерфейсов (кнопка «В корзину» справа).
- **Кейс Кинопоиска (2015):** Редизайн вызвал негодование — команда недооценила привычку. Пользователи привыкли к паттернам и не могли найти привычную работу «найти что посмотреть».
- **Привычка как барьер переключения:** Даже если наш продукт лучше, у человека сформирована привычка к текущему решению. Это мощный барьер.
- **Как учитывать привычку:** При изменении продукта сохранять привычные паттерны. При запуске нового продукта — понимать, от каких привычек нужно отказаться.

**Концепции / инструменты:**
Привычка, Нейросети автоматического поведения, Барьер переключения, Consideration Set и привычка, Редизайн и ожидания
